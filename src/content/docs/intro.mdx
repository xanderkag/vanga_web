# Что это и как работает

**Doc Parser** (внутреннее имя — Big Brother) превращает PDF-файлы и сканы в
структурированный JSON. На входе — счёт, накладная, акт или договор в виде
файла; на выходе — типизированные поля, готовые к загрузке в 1С, ERP или CRM,
с флагами доменной валидации.

<Callout type="info" title="Коротко">
  Отправляете файл одним POST-запросом, получаете <code>job_id</code>, а готовый
  результат прилетает вебхуком (или забираете опросом статуса). Никакого
  ручного разбора.
</Callout>

## Архитектура

Сервис состоит из двух частей:

- **doc-service** — публичный API, очередь заданий, хранение, классификация,
  типизированные парсеры (regex + Zod-схемы) и доставка вебхуков. Это та точка
  входа, с которой вы интегрируетесь.
- **inference-service** — внутренний слой моделей: OCR изображений
  (vision-LLM), извлечение полей по схеме и доменная валидация. doc-service
  обращается к нему сам; напрямую вы с ним не работаете.

Такое разделение позволяет менять модели и провайдеров без изменения
публичного контракта API.

## Что на входе и на выходе

| | |
|---|---|
| **Вход** | PDF, JPG, PNG (одно- и многостраничные) |
| **Выход** | JSON: тип документа, извлечённые поля, `confidence`, флаги валидации |
| **Доставка** | вебхук на ваш `webhook_url` или опрос `GET /api/v1/jobs/{id}` |

## OCR-каскад

Текст из документа извлекается каскадом движков — каждый следующий
подключается, если предыдущего недостаточно:

1. **pdf-text** — прямой текстовый слой PDF (самый быстрый и точный путь);
2. **tesseract** — локальный OCR для сканов без текстового слоя;
3. **vision-llm** — распознавание через модель, когда скан «сложный»;
4. **yandex** — облачный OCR как финальный резерв.

После распознавания документ классифицируется, поля извлекаются по схеме типа,
а затем проходят доменную валидацию (ИНН/КПП, НДС, госномера).
