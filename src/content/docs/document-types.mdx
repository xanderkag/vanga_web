# Типы документов

Сервис распознаёт **30 встроенных типов** документов — от бухгалтерских (счета,
счета-фактуры, УПД, акты) до транспортных и полного **ВЭД-пакета** (Commercial
Invoice, коносамент, таможенная декларация, сертификаты происхождения и
соответствия). Шесть базовых (`invoice`, `factInvoice`, `UPD`, `TTN`, `CMR`,
`AKT`) — это типизированные парсеры со строгой Zod-схемой и regex-извлечением;
остальные обрабатываются настраиваемым LLM-extract. Колонка **«Позиции»**
показывает, где сервис извлекает таблицу строк (товары/услуги/грузы) —
ключевой признак для счетов, накладных и спецификаций.

## Фазы парсера

- **Phase 1 · regex + LLM** — для типов со стабильной структурой. Сначала
  работает быстрый regex-парсер, LLM подключается там, где regex неуверен.
  Результат ложится в типизированную схему.
- **Phase 2 · LLM-extract** — извлечение полей по описанию схемы целиком
  моделью. Применяется к типам с вариативной вёрсткой; для длинных документов
  (200+ позиций) включается двухпроходный режим (header + батчи позиций).

## Зрелость типов

| Бейдж | Что значит |
|---|---|
| `stable` | **Production** — обкатан, типизирован, есть проверки качества. Заявляем точность. |
| `beta` | Работает в проде, активно дорабатываем по реальным данным. Подходит для пилотов. |
| `experimental` | **По запросу** — тип заведён, распознавание есть, качество не замерено. Включаем под конкретного клиента. |

<Callout type="info" title="Кастомные типы">
  Помимо встроенных, администратор может завести собственный тип в реестре
  (Document Type Registry): задать ключевые слова классификации, схему вывода и
  валидаторы. Новый тип подхватывается без перезапуска сервиса.
</Callout>

## Доменная валидация

Для извлечённых полей выполняются проверки: контрольная сумма **ИНН/КПП**,
формат **госномеров** ТС, **консистентность НДС** и сходимость сумм. Результат
проверок — в разделе [Ошибки и валидация](/docs/errors).
